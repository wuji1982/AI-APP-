本仓库采用多语言、多前端工程的依赖管理方式，各子项目各自维护独立的依赖清单，无统一的顶层锁文件或私有注册表配置。

后端（Python）
- 依赖声明文件：backend/requirements.txt
- 包管理器：pip（通过 docker-compose.yml 中 build: ./backend + Dockerfile 构建镜像时安装）
- 版本策略：全部使用 == 精确锁定版本，包括 FastAPI、SQLAlchemy、Celery、LangChain、OpenAI 等核心库
- 关键依赖分组：Web 框架（FastAPI/Uvicorn）、ORM（SQLAlchemy+asyncpg）、认证（python-jose/passlib）、异步任务（Celery+RabbitMQ）、缓存（Redis+aioredis）、对象存储（MinIO）、AI Agent（LangChain/LangGraph/OpenAI）、工具（httpx/python-dotenv）
- 无 lock 文件：未生成 requirements.lock 或 poetry.lock，依赖版本完全由 requirements.txt 控制

前端 Web 管理端（Vue3 + TypeScript）
- 依赖声明文件：frontend/web-admin/package.json
- 包管理器：npm/yarn/pnpm（脚本定义 dev/build/preview，基于 Vite）
- 版本策略：使用 ^ 语义化版本范围（如 vue: ^3.4.0），允许小版本升级
- 依赖分类：运行时依赖（vue、pinia、element-plus、axios、echarts）与开发依赖（vite、typescript、vue-tsc）分离
- 无 lock 文件：未发现 package-lock.json / yarn.lock / pnpm-lock.yaml

移动端（uni-app）
- 配置文件：frontend/mobile-app/manifest.json（仅应用元数据，非依赖声明）
- 依赖管理：未见 package.json，可能通过 uni-app CLI 隐式管理或使用 HBuilderX 内置依赖解析
- 状态：该目录结构不完整（缺少 package.json），依赖管理处于缺失状态

基础设施依赖（容器编排）
- 编排文件：docker-compose.yml
- 服务镜像版本：PostgreSQL 16-alpine、Redis 7-alpine、RabbitMQ 3-management-alpine、MinIO、Nginx alpine
- 健康检查：对 PostgreSQL 使用 pg_isready 进行就绪探测
- 卷持久化：pgdata、redisdata、miniodata 三个命名卷

设计决策与问题
- 缺乏统一锁文件：前后端均无 lock 文件，可能导致不同环境构建结果不一致
- 版本策略不统一：后端严格锁定、前端宽松范围，存在升级风险差异
- 移动端依赖缺失：uni-app 工程缺少 package.json，无法被标准 npm 生态识别
- 无私有源配置：未发现 .pypirc、.npmrc 或 Docker registry mirror 配置，默认使用公网 PyPI/npm 源
- 无依赖更新机制：未发现 Dependabot、Renovate 等自动化更新配置
