模块采用典型的分层架构：
- `app/main.py` 通过 `lifespan` 钩子在启动时调用 `Base.metadata.create_all`（生产走 alembic），注册 CORS 中间件并按 `/api/v1/{domain}` 前缀挂载各业务路由。
- `app/api/v1/` 下每个领域一个 router 文件（auth/user/product/group_buy/contribution/points/coupon/store/admin），仅做请求解析与参数校验，不写业务逻辑。
- `app/services/` 对应每个 API 的 Service 类，封装事务性业务流程（如 `GroupBuyService.join_group_buy/settle_session`），直接操作 `app/models/` 下的 SQLAlchemy 模型并通过 `get_db()` 依赖注入 `AsyncSession`。
- `app/database.py` 使用 `sqlalchemy.ext.asyncio` 创建 `engine` 与 `async_sessionmaker`，暴露 `get_db` 作为 FastAPI 依赖；`alembic/versions/` 存放迁移脚本。
- `app/tasks/` 以 `celery_app.Celery` 为入口，按业务拆分 task 文件（contribution/dividend/group_buy/store_rank），由 RabbitMQ 作 broker、Redis 作 result backend。
- `app/agents/` 基于 LangGraph 实现 Agent 状态机，`base_agent.BaseAgent` 定义 `execute/should_continue/run` 抽象接口，`agent_orchestrator` 负责编排多 Agent 协作。
- 配置集中在 `app/config.Settings`（pydantic-settings），从 `.env` 加载数据库、Redis、Celery、MinIO、JWT、以及大量拼团/分润/贡献值等业务常量，所有 Service 通过 `settings.*` 读取规则。
- 依赖方向严格单向：API → Service → Model/Database，Task 与 Agent 可跨层调用 Service，但 Service 不反向依赖 API。
